# Sand Together — 系统架构

> 本文件定义**系统分层、数据流和技术选型理由**。
> 变更技术选型前必须在本文件说明理由，并在 `CLAUDE.md` 中确认是否影响硬约束。

---

## 系统分层结构

核心三层，硬件端为可选扩展层：

```
                        ┌─────────────────────────────────────────────┐
                        │  Layer 1 · Web UI（纯渲染，无权威物理）       │
                        │  纯 HTML/CSS/vanilla JS，GitHub Pages 托管   │
                        │  发输入(计数/交互) → 收服务端网格 → 渲染      │
                        └──────────▲──────────────────▲───────────────┘
                                   │ Tauri 事件        │ WebSocket(wss)
                                   │ 'keycount'        │ 输入↑ / 网格状态↓
          ┌────────────────────────┴──────┐  ┌─────────┴──────────────┐
          │  Layer 2 · Tauri 外壳 (Rust)   │  │  Layer 3 · 后端服务      │
          │  透明/无边框/置顶 原生窗口      │  │  (VPS · Node + ws)       │
          │  加载 Pages URL               │  │  每房间一个权威物理模拟   │
          │  全局输入统计 (rdev)           │  │  持有 grid = 唯一真相     │
          │  → emit keycount 给 webview   │  │  增量广播 + 持久化存盘    │
          └──────────────┬────────────────┘  └─────────────────────────┘
                         │ 可选：ticks 输入源
          ┌──────────────┴────────────────┐
          │  Layer 4 · 硬件端（可选）      │
          │  专用显示设备 / 传感器          │
          │  作为额外的输入源             │
          │  或作为独立的可视化显示器       │
          │  接口方案待定（见 hardware.md） │
          └───────────────────────────────┘
```

## 数据流

### 单机闭环（Stage 1）

```
你打字/点击
  → Rust (rdev) 监听 KeyPress/ButtonPress → 累加 count
  → emit('keycount', count) 给 webview
  → webview：生成对应颜色沙粒，执行本地物理模拟
  → 每 25 次写入本地文件（app_data_dir/keycount.txt），重启恢复
```

### 多人房间（Stage 2）

```
本机 count
  → 前端节流（10fps）→ ws.send({ type:'progress', ticks })
  → 房间 DO 合并 → 广播 { type:'state', players: { id: { name, color, ticks } } }
  → 各端：根据对应玩家 ticks 决定该用户出沙速率
  → 沙粒轨迹各自本地计算（不同步物理状态）
```

**新玩家加入 / 重连：读快照，纯渲染**

```
新玩家加入房间
  → 服务端立即下发当前完整 grid 快照（+ 玩家档案）
  → 客户端直接渲染这份快照（不重放、不本地模拟）
  → 之后持续接收增量广播，与所有人保持一致
```

**关键决策（2026-06-21 转向）：物理在服务端权威计算**
- 原因：客户端各自本地模拟会因随机滑落而发散，无法保证"所有人看到一致"，且已落定的沙没有单一权威、难以作为房间存档
- 取舍：服务端是唯一真相、客户端纯渲染 → 完全一致、整块画布持久化；代价是后端从无状态中继变为常驻有状态服务（跑物理 + 带宽），需自建 VPS
- 带宽：增量广播（只发变化单元）；稳态下只有堆顶活跃前沿在变，已落定部分发一次即不动
- **被替换**：旧的"物理本地化 + 只同步 ticks + 冻结地层客户端上报"整套作废

> ⚠️ **后续演进（已上线）**：本段「客户端纯渲染、只收 grid 增量」是 2026-06-21→22 那一代的描述；之后转向**确定性帧同步（lockstep）**——客户端**也**跑同一份确定性 sim（`/sim.js`），自己的沙零延迟预测，`patch` 降为老客户端兼容路径。这并不推翻"服务端是唯一真相 + 裁决者 + 持久化"（仍成立），只是客户端从「瘦」变「胖」。**最新模型以本文「同步模型」节为准**，上面的数据流 / 分层图按那一代理解。

### 房间模型

- **进入方式**：玩家默认创建一个新房间并开始记录；也可以通过邀请或输入已有房间号，直接加入已存在的房间
- **房间容量**：最多 4 名玩家，地位完全对等，没有"房主"或"访客"之分
- **房间生命周期**：房间数据由服务端持久化（grid 快照 + 玩家档案存盘），只要不是全部成员主动退出/弃用，房间及其数据永久保留——不依赖是否有人在线
- **存档解耦（ARK 式，2026-06-24）**：玩家数据与世界数据分两套独立存储——
  - **世界档案**（每 `roomId` 一份）：grid + bands + **成员名单** `members:{pid:{color,ticks,contributionTicks,joinedAt}}`（不再存 name）。
  - **玩家档案**（全局，每 `playerID` 一份）：`name` + `skills`（天赋，预留）+ `lifetime`（全局累计 = 未来买技能的资源池）+ `worlds`（加入过哪些世界，双向成员索引）。**全局角色**：技能/天赋归玩家本人，加入任何世界都生效。
  - 两者经 playerID 关联；服务端 `rosterForWire()` 把成员(color/ticks)+档案(name) 合成回旧名册形状下发，**线协议不变**。详见 `backend.md`。
- **玩家数据模型**：`playerID`（持久标识，前端存进 localStorage 复用；当前即玩家档案的 key，**暂沿用设备级 UUID，账号/登录留后**——数据结构已为其预留，将来加账号无需迁移）+ `color`（每世界内分配，未来可手动选择）。
- **布局**：1～4 名玩家在视觉上居中对称排列（详见 `frontend.md`），位置对应"沙粒出口"，不对应历史沉积区域——人数变化只影响新沙粒从哪里开始落下，已冻结的历史像素不受影响

### 硬件端接入（待定）

硬件端只是另一种 ticks 输入源，接入后数据流与本机键盘输入等价：

```
硬件传感器 / 设备
  → 通过待定接口（USB HID / BLE / 串口 / WebSocket）提供 ticks
  → 软件端像对待本机 keycount 一样处理
  → 或作为独立显示器，接收并渲染房间状态
```

具体接口方案见 `doc/hardware.md`（待确认后填充）。

## 同步模型（确定性帧同步 / lockstep，已上线）

> 「物理在哪算 / 同步什么」是**会演进的架构选择**。本节是它的**单一权威记录**；具体线协议「档案」归 `backend.md`。

**当前实现 = 确定性帧同步（lockstep）+ 自沙预测 + 快照兜底**（取代了早期「服务端跑物理、客户端只渲染 `patch`」）。CA 抽进共享 `/sim.js`，**服务端与客户端跑同一份**：平时网络只传**输入**（`frame` 有序事件流），客户端本地即时重现网格、自己的沙**零延迟预测**；服务端在背后裁决输入顺序、持久化，并用周期 checksum 让客户端**自愈**。延迟消失的真正原因是**本地也在跑那份确定性模拟**——打字后本地立刻出沙、不等服务器往返。

> 兼容：未加载 `/sim.js` 的老客户端（或加 `?patch`）退回「收逐格 `patch` 纯渲染」；服务端只对这些非 lockstep 连接发 `patch`/`band`。整房可用 `SAND_EMIT_FRAMES=0` 退回纯 patch。

为什么本项目能 lockstep（多数物理游戏不能）：falling sand 是**整数网格上的元胞自动机**，格子交换确定，只要种子固定、随机消费顺序一致，就**没有浮点漂移**——那个逼 3D 物理游戏走 client-prediction + rollback 的拦路虎，在这里天然不存在。

### 确定性纪律（硬约束，破一条即发散）

- **带种子 PRNG**（mulberry32，整数运算），所有节点同种子；随机**消费顺序一致**。
- **固定 tick**（50ms）；CA 内**只用整数**，禁浮点 / 系统时间 / 平台 API。
- 同 tick 多玩家沙粒**确定处理顺序**（按 playerId 排序）。
- **每 60 tick 做 checksum**（FNV-1a 网格哈希），随 `frame` 下发、客户端比对自愈。
- CA 收敛成**唯一一份源码** `/sim.js`（服务端 `require` + 网页 `<script src>`），两份代码差一行即漂——无构建步骤约束下保确定性的关键。

### 消息协议（已实现；线档案见 `backend.md`）

| 类型 | 方向 | 作用 |
|---|---|---|
| `input` | 上行 | 玩家累计击键数 / 交互事件，极小载荷 |
| `frame` | 下行 | 服务端裁决后的逐 tick 有序事件流（lockstep 主力）；每 60 tick 附 `chk` 网格哈希 |
| `snapshot` | 下行 | 全量权威状态（含 sim 状态 rng/frame/queues…）；**仅**热加入 / 重连 / 自愈 |
| `resync` | 上行 | 客户端 checksum 不符 → 要一份新 snapshot 拉回 |
| `patch` / `band` | 下行 | 逐格增量 / 归档带——**仅**发非 lockstep 客户端（兼容路径） |

> 不止击键：`spout`/`join`/`leave`/`reset`/`flood`/`pour` 都改确定性模拟，故都钉 tick 进 `frame`。模型 = **grid 是 (种子 + 有序事件流) 的纯函数**，服务端是时钟 + 定序权威。

### 延迟策略（已实现 = 自沙预测）

- **自己的沙**：本地**预测**——`renderSim` = 克隆权威镜像 `authSim` + 你「已打字未确认」的沙 + 快进 `PREDICT_LEAD` tick，立即渲染。因始终从 authSim 重算，漂移天然限制在你最新几粒（合堆时至多 ≤1px 修正；归档保持权威-only，无瞬跳）。
- **别人的沙**：走 playout 缓冲 `FRAME_BUFFER`（=2 tick），吸收网络抖动。
- `authSim`（权威镜像）只由 `frame` 推进、逐位跟随服务端；`renderSim`（渲染源）按需从 authSim 重算（脏标记：authSim 推进 / 你打字时，而非每帧白算）。

### 快照的角色（兜底，不是主力）

仅在 ①热加入 ②自愈（checksum 不符 → `resync`）③断线重连 时下发；复用「新加入收当前快照、不重放历史」逻辑。这正是学界「混合架构」：稳态跑 lockstep，re-sync / hot-join 用快照。

### 权威与持久化

服务端是**输入定序的唯一裁决者** + 唯一权威存档（无限期保留，全员弃用才回收）。活跃区瞬态小分歧由 checksum 自愈兜住；冻结地层（成就 freeze，**功能待建**）落地时将天然由这套服务端定序保证全员一致，无需额外机制。

### 已定参数 / 仍开放

- **已定（已上线）**：`FRAME_BUFFER`=2、`PREDICT_LEAD`=B+1、checksum 每 60 tick、`resync` 冷却 4s、离线 = 丢弃 + 重连快照、延迟模型 = **自沙预测（A3）**。
- **仍开放**：`PREDICT_LEAD` / `FRAME_BUFFER` 真机（大陆→香港）手感微调；成就 freeze 的多端裁决留到该功能落地（架构已覆盖，见上）。

## 技术选型

### 为什么 Web + Tauri

产品主体是数据可视化 UI，Web 做 UI 最快、迭代成本最低。透明叠加层 + 全局输入在任何路线都需要原生能力，Tauri 的原生适配面最小、最轻。其他技术路线（如游戏引擎）留给后续阶段按需评估，不在当前阶段排除。

### 为什么 VPS + Node 常驻服务（权威物理）

- 服务端权威物理需要一个**常驻、有状态**的进程持续跑模拟循环 + 在内存里持有 grid——这正是 Serverless（CF Workers/DO 按需唤醒、CPU 时限）不擅长的
- Node 直接复用前端那套 JS 物理代码（搬迁为主，不重写）
- `ws` 库做 WebSocket，单进程内"每房间一个模拟"与房间模型 1:1 对应
- 部署在海外位置的 VPS（大陆厂商香港/新加坡节点亦可），大陆与海外都可连，且绕开 `*.workers.dev` 被墙

### 历史：曾用 Cloudflare Workers + Durable Objects（已弃用）

Stage 2 早期后端是 CF Workers + DO（只中继整数 ticks、不跑物理；曾因 PartyKit 共享域名 `partykit.dev` 撞 Cloudflare 子域名上限而改用底层 DO）。2026-06-21 转向服务端权威物理后，这套（无状态中继 + 客户端本地物理）整体作废，由常驻 Node 服务取代。

### 为什么 rdev

跨平台全局输入监听，能逐次捕获 `KeyPress`/`ButtonPress` 用于计数，无需记录内容。

## 已知风险

- **rdev 焦点夺键（Tauri #14770）**：rdev 在 Tauri 进程内时，Tauri 窗口获得焦点会收不到键盘事件。缓解：`device_event_filter(Never)`；终极方案是独立监听进程。
- **Tauri v2 能力权限（静默失败陷阱）**：webview 里的 JS 调用窗口几何 API（`setSize`/`setPosition`/…）必须在**该窗口的能力文件**（`src-tauri/capabilities/*.json`）里授予对应 `core:window:allow-*`，否则 Promise **异步静默 reject**、调用无效（控制台无报错）。本项目按窗口分：`default.json`=main、`menu.json`=菜单弹窗。菜单的内容自适应 `setSize` 曾因缺 `core:window:allow-set-size` 一直失败、窗口卡死并裁掉展开内容。排查真机 WebView2 用 `WEBVIEW2_ADDITIONAL_BROWSER_ARGUMENTS=--remote-debugging-port=N` + CDP（浏览器测不出权限层）。
- **大陆可达性**：转向 VPS 后由海外位置节点（香港/新加坡等）承载，大陆与海外均可连，免 `*.workers.dev` 被墙问题。需自备域名/直连 IP + 证书（wss）。
- **服务端带宽/算力**：常驻进程跑物理 + 增量广播；高频多人时需关注 VPS 带宽与 CPU。增量广播 + 限帧（~15–30fps）控制成本。
- **杀软告警**：全局输入钩子像键盘记录器，Windows Defender/SmartScreen 可能提示，自机放行即可。
- **物理性能**：活跃粒子上限 5000 颗（见 `frontend.md`），超出自动压缩，避免 CPU 占用过高导致用户卸载。
- **网络延迟下的渲染**：服务端权威下不再有多端物理分叉；但客户端渲染依赖广播到达，弱网时表现为画面卡顿/跳帧（而非发散）。可在客户端做轻量插值缓解（非必须）。

## 阶段计划

### Stage 1 · 单机（已完成）

- Tauri v2 外壳，透明/无边框/置顶，加载 Pages URL
- rdev 全局输入统计，emit 到 webview
- 本地 canvas 像素沙基本运行（沙粒生成 + 重力 + 堆积）
- 本地计数持久化

### Stage 2 · 多人 + 服务端权威 + 桌宠 App（已完成 2026-06-22）

服务端权威物理转向完成并上线：

- 常驻 Node + ws 后端，每房间一个权威 falling-sand 模拟（协议见 `backend.md`）；物理从前端搬到服务端，前端改为纯渲染（发输入、收 grid 增量、画）
- 增量广播 + 整张网格服务端持久化；新玩家加入收完整快照，同房间所有人**逐帧一致**
- 部署：阿里云香港轻量 VPS + Caddy 自动 TLS，`wss://sandtogether.indiegames.design`，VPS 一站式（`GET /` 托管页面 + ws）；弃用 GitHub Pages 与旧 CF worker
- 桌宠 App：Tauri 加载 VPS 页面（透明置顶 + 全局 keycount + wss 同步），已出可分发安装包（`.exe`/`.msi`）

### Stage 3 · 产品化（进行中）

- 同步升级为**确定性帧同步（lockstep）+ 自沙预测 + checksum 自愈**：✅ **已实现并上线**（客户端跑共享 `/sim.js`、自己的沙零延迟；`patch` 降为老客户端兼容路径，省带宽；模型见本文「同步模型」）。剩余：`PREDICT_LEAD`/`FRAME_BUFFER` 真机手感微调
- 沉积快照的视觉压缩与展开：✅ **已实现**（服务端把活动网格底部老沙折叠成归档带 `band`、整体下移腾顶；客户端把归档按**真实高度展开**渲染，滚动到附近即自动展开成连续历史，压缩对用户透明，机制见 `backend.md`）。归档**逐像素无损**（band 存真实像素 `cells`，滑下去看到的和当时一模一样）；体积压缩（RLE/gzip）是明确的第二步、暂未做。剩余：超深历史的二级压缩
- 成就系统（里程碑触发成就物件落入并永久封存）
- 游戏化系统（赌注、暴击、非操作性累积等，详见 `game-design.md`）
- 双形态（小窗叠加层 ↔ 展开面板）
- 皮肤/主题系统（利用现有 CSS 变量）
- 硬件端接入（专用显示设备或传感器输入，见 `hardware.md`）
